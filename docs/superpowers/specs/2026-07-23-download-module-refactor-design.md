# 下载模块整体重构设计

## 1. 目标

将 `media_downloader.py` 从同时承担启动、扫描、排队、传输、包编排和结果汇总的超大脚本，收敛为稳定的兼容入口与应用组装层。下载模块按“资源包生命周期”和“单文件传输结果”分层，保持现有 Web、Bot、频道库和测试调用兼容。

本次重构解决以下问题：

- 入口文件职责过重，启动逻辑与业务逻辑互相依赖。
- “一个下载任务”在提交任务、资源包和单文件三个粒度上含义不清。
- 多频道同名资源包可能因仅使用标题而混淆。
- 多关键词命中多个资源包时缺少统一、可证明的调度顺序。
- 下载失败、中断、进度迟滞、上传失败和上传权限错误缺少一致的生命周期结论。
- 下载与上传耦合，导致上传失败时容易错误地重新下载。
- 磁盘空间只在写入时被动失败，无法决定何时允许下一个资源包开始。

## 2. 已确认决策

- Telegram 客户端继续使用 Pyrogram，不迁移到 Telethon。
- 参考 Telethon 的可靠性语义处理 FloodWait、临时网络错误、重连等待和持久化错误，但不引入第二套 Telegram 客户端。
- 提交任务是用户或规则创建的一次批次；资源包是 FIFO 排队和磁盘预留单位；文件是实际传输和结果记录单位。
- 所有自动触发资源包按创建时间进入全局 FIFO，不设置关键词优先级，不允许后来的任务插队。
- 同一关键词命中的多个包、多个关键词命中的多个包，都先去重为唯一包快照，再按入队时间和包内时间顺序执行。
- 下载和上传是两个独立阶段，共用一个资源包生命周期。
- 资源包开始下载前按快照中的已知总大小预留磁盘；默认必须额外保留至少 3 GiB 可用空间。
- 磁盘预留从包获准开始一直保持到下载、上传和清理结束。
- 包内单文件失败后继续处理剩余文件，最终允许以 `completed_with_errors` 结束。
- 下载失败或中断产生单文件错误记录；显式重试下载时从头下载该文件，不保留不可验证的临时分片。
- 上传失败或上传权限错误保留已下载文件，只进入上传重试列表；重试时仅执行上传，不重新访问 Telegram。
- 用户执行显式清理时直接删除上传失败所保留的本地文件，并释放对应生命周期占用。
- 已成功下载或已变为 `outdated` 的包不参与自动重复下载；手动明确提交下载不受该自动保护终止。
- 关键词监控组只匹配数据库中的当前资源包索引。保留“指定关键词 -> 已扫描频道资源包 -> 触发下载”链路，不做实时频道监控、历史轮询或 Discord 通知。
- 关键词不从 `config.yaml` 读取，由数据库和 Web 页面统一管理。
- 资源包升格为跨频道聚合接口；包仍通过 `library_id` 关联来源频道，搜索和选择不要求先进入单个频道。

## 3. 术语与身份

### 3.1 提交任务

一次手动操作或一次自动规则触发创建的不可变批次。任务负责保存来源、创建时间、幂等键、选中的包快照和总体进度，不直接作为磁盘预留单位。

### 3.2 资源包

一个频道索引中由现有包规则得到的有序媒体集合。资源包是：

- FIFO 准入单位；
- 磁盘容量预留单位；
- 下载与上传共享生命周期单位；
- `completed`、`completed_with_errors`、`upload_failed` 等包级结论单位。

稳定身份不得只使用标题。频道库资源包身份至少由：

`library_id + package_id + index_revision`

组成。下载批次保存不可变的 message ID 集合、标题和 revision 快照。不同频道的同名包、同一频道不同起点的同名包、重建索引后的旧 revision 均不会混淆。

### 3.3 文件

资源包中的一个 Telegram message/media。文件是下载、上传、进度、错误和重试的最小记录单位。单文件失败不立即终止整个包。

## 4. 模块边界

### 4.1 `media_downloader.py`

只保留：

- CLI 启动；
- 将 `media_downloader` 导入别名绑定到 `module.download_entry`，保持旧调用方和
  monkeypatch 目标兼容。

入口不得再包含 Discord webhook、实时频道监听或兜底轮询实现。

### 4.2 `module/download_entry.py`

负责兼容导出和依赖组装：

- 旧调用方需要的稳定公开符号和包装函数；
- 全局 `Application`、Pyrogram client 和进程级队列的组装；
- 配置检查；
- 向下载子模块显式传入当前依赖。

### 4.3 `module/download_transfer.py`

负责单文件下载：

- 媒体类型与格式判断；
- 文件名和目标路径生成；
- 临时文件下载、进度观察、完成大小校验和原子移动；
- Pyrogram 异常分类、FloodWait 等待、临时错误重试和迟滞中止；
- 下载成功、跳过、失败的文件级结果。

该模块不决定资源包顺序，不创建频道库批次，不负责上传重试。

### 4.4 `module/download_lifecycle.py`

负责单文件共享生命周期：

- `run_download_phase` 只写下载阶段证据；
- `run_upload_phase` 只使用本地完成文件，不回调下载；
- `run_file_lifecycle` 只负责两个阶段的顺序、报告和统计收尾。

### 4.5 `module/download_queue.py`

负责进程内文件队列：

- 固化入队时的命名快照和 Telegram activity intent；
- worker 领取与释放；
- 兼容旧二元、三元队列项；
- 把单文件工作交给 transfer；
- 保证取消时释放 intent，避免扫描永久饥饿。

### 4.6 `module/package_download.py`

负责资源包执行：

- 对选中包按快照顺序串行执行；
- 为包设置命名上下文；
- 包内继续处理剩余文件；
- 根据每个预期 message ID 的持久化结果生成包级结论；
- 调用包开始/结束回调；
- 不用父任务累计计数推断单个包结果。

### 4.7 `module/download_admission.py`

负责磁盘准入：

- 按资源包入队顺序 FIFO 等待；
- 用 `free - active_reservations - requested >= minimum_free` 判定是否准入；
- 不允许后来的小包绕过前面等待的大包；
- 取消等待时移除队列项；
- 生命周期结束后只释放一次。

### 4.8 `module/download_runtime.py`

负责应用运行期：

- 周期进度刷新；
- 普通聊天任务装配；
- Pyrogram client、Web、Bot、频道库服务和 worker 的启动/停止；
- 关闭时先停止业务服务，再停止 Telegram client，并取消本进程创建的后台任务。

业务模块不得自行创建第二事件循环。

## 5. 调度与顺序

### 5.1 全局包顺序

资源包准入使用持久化批次创建时间作为主顺序，批次内使用包 ordinal 作为次顺序，最终以唯一 attempt ID 消除并列：

`batch.created_at, package.ordinal, attempt_id`

关键词仅解释触发原因，不参与优先级。一个包同时命中多个监控组时，只创建一个包 attempt；每个监控组分别保存该组实际命中的关键词、包 revision、批次和任务状态。

### 5.2 包内文件顺序

按包快照保存的 ordinal/message ID 顺序入队。允许多个文件 worker 并发执行，但资源包完成必须等待快照内所有文件进入终态。若需要严格串行文件传输，应通过 worker 数配置实现，不在包模型中引入第二套优先级。

### 5.3 手动任务

手动提交可绕过“成功/outdated 不自动重复”的过滤，但不能越过已获得磁盘预留并正在执行的包。手动和自动包统一进入 FIFO，避免两套调度规则。

## 6. 生命周期

资源包状态按以下方向推进：

`queued -> waiting_disk -> downloading -> uploading -> cleanup -> terminal`

终态包括：

- `completed`：所有预期文件下载及所需上传成功，或是可证明的已完成文件跳过。
- `completed_with_errors`：包内至少一个文件失败，但其余文件已继续处理完毕。
- `upload_failed`：下载完成，但至少一个需要上传的文件上传失败或权限不足，本地源文件被保留。
- `failed`：包无法开始或没有任何可继续的有效文件。
- `cancelled`：用户停止且剩余文件未继续执行。
- `outdated`：历史成功 revision 与当前索引 revision 不同；这是自动下载排除状态，不代表当前 revision 已下载。

`upload_failed` 是可恢复终态。上传重试开始时恢复为活动状态，成功后进入 cleanup/terminal；再次失败仍回到 `upload_failed`。

## 7. 容错规则

### 7.1 下载失败

- 记录 message ID、阶段、稳定错误码和可读摘要。
- 删除本次未完成的临时文件。
- 继续资源包内其他文件。
- 显式下载重试从头开始。

### 7.2 下载中断

- worker 取消必须释放 Telegram activity intent。
- 已有完整且通过大小校验的目标文件可作为完成证据。
- 未完成临时文件直接删除，不做断点续传。
- 包根据已完成文件和剩余取消文件进入 `completed_with_errors` 或 `cancelled`。

### 7.3 下载进度迟滞

- 每个 message 记录最后进度时间和字节数。
- 超过配置迟滞窗口没有字节增长时取消当前 Pyrogram 下载调用。
- 迟滞按可重试下载失败处理；达到次数上限后记录稳定错误并继续包内剩余文件。
- watchdog 结束时必须清除 message 级内存状态。

### 7.4 FloodWait 与临时网络错误

- FloodWait 使用 Telegram 返回的等待秒数，不进行忙循环。
- 可恢复网络错误按有限次数退避重试。
- 等待原因、截止时间和最近错误写入持久化任务/attempt 状态，以便重启和 Web 展示。
- 文件引用过期允许重新获取原 message；重取失败记为该文件失败。

### 7.5 上传失败和权限错误

- 下载成功证据与上传结果分开保存。
- 上传失败不把文件状态回退为下载失败。
- 本地文件加入上传重试列表，并保持磁盘预留。
- 上传重试只读取本地文件并调用上传适配器。
- 文件缺失时记录 `upload_source_missing`，不得静默重新下载。
- 显式清理校验路径位于配置的保存目录内，再删除文件并记录 `upload_source_removed`。

### 7.6 未知大小

首版不对未知大小资源包进行猜测性预留。包以 `unknown_package_size` 安全失败并继续调度后续包。以后如增加受控上限或探测机制，必须另行修改本 spec。

## 8. Pyrogram 适配规范

- 所有 Telegram 调用集中在适配边界，业务层不捕获无法稳定识别的字符串错误。
- 优先按 Pyrogram 异常类型分类：FloodWait、文件引用过期、RPC 权限错误、临时网络错误和不可恢复请求错误。
- FloodWait 秒数兼容当前 Pyrogram 暴露字段，但只在适配层处理版本差异。
- 不在失败后并行重建多个 client，不共享另一个事件循环上的 session。
- 重取 message 使用原 chat ID 和 message ID，重新验证返回对象及媒体存在性。
- 下载取消与 client shutdown 都必须可等待，避免遗留后台传输。

## 9. 兼容性

- `media_downloader` 继续导出当前 Web、Bot、测试和频道库使用的函数与数据类型。
- 兼容层只做依赖组装和参数转发，不复制业务逻辑。
- 现有任务数据库和频道库 schema 只做已确认的加法变更；不重写历史数据。
- 现有 Pyrogram session、配置文件和启动命令保持不变。
- `monitor` 配置仅作为旧配置兼容输入，不再启动实时监控或 Discord webhook；频道库关键词规则不从任何配置项读取。

## 10. 验证标准

- `media_downloader.py` 不再包含旧实时监控/Discord 实现，`main()` 仅负责组装。
- 新模块具有单元测试，覆盖 FIFO、取消释放、3 GiB 下限、未知大小、包内部分失败、迟滞、上传失败保留、仅上传重试和清理。
- 不同频道同名包使用不同稳定身份。
- 同包多关键词只创建一个 attempt；多包严格按持久化创建顺序执行。
- 已成功/outdated 包不会被自动重复提交，手动提交仍可运行。
- 现有 `media_downloader` 公开导入和 monkeypatch 测试继续通过。
- 定向下载生命周期测试和完整测试套件通过。
- `py_compile` 与 `git diff --check` 通过。
